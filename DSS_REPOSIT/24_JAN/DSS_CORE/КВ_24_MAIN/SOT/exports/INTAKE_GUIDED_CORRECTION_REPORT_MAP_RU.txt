INTAKE ↔ GUIDED CORRECTION ↔ CLIENT REPORT (RU)

Цель: единая управленческая карта соответствий без технички.

Формат строки: Intake → Что закрывает → Куда попадает → Если FAIL

Q1. Основная проблема (1–2 предложения)
- Закрывает: формулировку решения и контекст
- В отчёте: Решение / Резюме / Decision Lock
- Если FAIL: нет ясного предмета решения → статус «Требует уточнения входа»

Q2. Владелец проблемы и зона контроля
- Закрывает: ответственность и исполнимость
- В отчёте: Владелец решения / План / Rule Owner
- Если FAIL: нельзя назначить ответственность → «Требует уточнения входа»

Q3. Жёсткие ограничения (время/бюджет/комплаенс)
- Закрывает: границы допустимых решений
- В отчёте: Цена ошибки / Правило / Границы
- Если FAIL: риск нелегитимного решения → «Требует уточнения входа»

Q4. Метрики успеха
- Закрывает: критерии решения
- В отчёте: Метрики и критерии успеха
- Если FAIL: нельзя проверить решение → «Требует уточнения входа»

Q5. Где происходит проблема (границы системы)
- Закрывает: область решения
- В отчёте: Структура проблемы / Границы
- Если FAIL: невозможно определить сферу решения → «Требует уточнения входа»

Q6. Главное противоречие
- Закрывает: конфликт, который решение должно снять
- В отчёте: Противоречие / IKR
- Если FAIL: нельзя построить решение → «Требует уточнения входа»

Q7. Ресурсы, которые уже есть
- Закрывает: реализуемость без новых затрат
- В отчёте: План / Решение / Механизм
- Если FAIL: решение не исполнимо → «Требует уточнения входа»

Q8. Горизонт результата
- Закрывает: реалистичность и планирование
- В отчёте: План исполнения / Метрики
- Если FAIL: нет реального плана → «Требует уточнения входа»

Обязательное правило:
FAIL не виден клиенту как PASS/FAIL. Клиент видит только:
- «Решение готово»
- «Требует обязательного уточнения входных данных»
- «Решение невозможно при этих условиях»
