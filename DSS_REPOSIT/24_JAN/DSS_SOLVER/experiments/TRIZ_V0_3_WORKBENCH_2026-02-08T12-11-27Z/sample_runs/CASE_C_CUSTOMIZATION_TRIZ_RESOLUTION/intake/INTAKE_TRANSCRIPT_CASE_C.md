Q1 [S00]: В чём тупик (1–2 предложения)?
A1: Крупный клиент требует кастомизацию ядра; без неё сделка может сорваться, с ней риски деградации продукта и роста поддержки.

Q2 [S01]: Кто принимает решение и чем реально управляет?
A2: CEO/Founder. Контроль: политика кастомизации, роадмап, бюджет dev/support, условия контракта.

Q3 [S01]: Жёсткие ограничения.
A3: Нельзя нанимать в ближайшие 2–3 месяца; нельзя ломать сервис текущим клиентам; нельзя создавать не поддерживаемый форк; решение нужно в 2 недели.

Q4 [S04]: Baseline и/или proxy‑диапазоны + дата валидации.
A4: PROXY baseline релизов: 2–3 релиза/месяц. Validation date: 2026-03-01.

Q5 [S12]: Thresholds (switch + kill/stop).
A5: Switch: если кастомизация превышает 20% ёмкости — переводить в отдельный контур. Kill: если нагрузка поддержки превышает +25% — прекращать кастомизацию. Validation date: 2026-03-01.

Q6 [S08]: Где разделить проблему (time/space/condition/part‑whole)?
A6: Разделить по условиям и по частям: кастомизация только для ограниченного сегмента и только в отдельном слое, не затрагивающем ядро.

Q7 [S08]: Принципы → действия (минимум 2).
A7: Принцип «разделение по условиям» → отдельный контрактный контур с SLA; Принцип «дробление» → модульный слой расширений вне ядра.

Q8 [S09]: Минимум 2 концепта и ссылка на принцип/разделение.
A8: Концепт 1: «Модуль расширений» (trace: дробление + разделение по частям). Концепт 2: «Контур кастомизации для сегмента» (trace: разделение по условиям).
